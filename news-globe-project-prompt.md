# 项目构想：地图化新闻/事件展示

## 背景
这是从"Global Times"(3D地球时钟)项目衍生出的独立想法，建议作为新项目单独开发，不要并入时钟项目。

## 核心想法
在地图/地球上，用不同颜色的星标标记各地正在发生的重大事件/新闻，鼠标悬浮时显示该地点相关新闻链接列表。星标颜色代表重要程度分级（如：非常重大 / 重大 / 中等）。

## 已知的关键难点（需要优先确认方案）

1. **数据源选型**
   - 直接爬取各大新闻网站存在版权和反爬风险，不推荐。
   - 推荐方向：RSS订阅聚合、新闻聚合API（如 NewsAPI、GDELT Project 等）。
   - GDELT 本身已经包含事件分类和地理信息，可能是最适合"事件分级+地理定位"这个需求的数据源，值得重点调研。

2. **地理定位（最大难点）**
   - 从新闻文本中提取"具体地点"并转换为经纬度（地理编码/NER），规则引擎在这一步能力有限。
   - 如果用 GDELT 等已结构化数据源，这一步可能已经被解决，应作为选型的重要考量。

3. **架构复杂度**
   - 一旦涉及"定时抓取 + 分级处理 + 数据存储"，项目就从纯前端静态站升级为需要后端/定时任务/数据库的应用，部署运维成本显著上升。
   - 需要明确：是否接受这个复杂度，还是先做轻量版本。

4. **分级逻辑**
   - 优先用规则引擎（关键词、来源权重、事件类型、影响范围等）而非 AI，成本低且可解释。
   - 如果用 GDELT，其自带的 Goldstein Scale / 事件强度评分可以作为分级基础。

## 两个可行的简化方向（如果完整版本风险太高）

### 方向A：历史/重大事件时间轴（零后端起步）
- 不做实时抓取，预先整理一份结构化数据集（地点+经纬度+重要程度+日期+链接）。
- 地理定位和分级在数据准备阶段人工/半自动完成。
- 前端依然做星标+悬浮列表展示，纯静态部署。
- 适合作为 MVP 验证交互效果，之后再考虑动态化。

### 方向B：地图驱动的地区新闻浏览
- 不做全网聚合和分级，用户点击/悬浮某城市时，调用新闻API按地区/语言拉取该地当前热门新闻。
- 跳过地理编码和分级两大难点，聚焦"探索某地此刻发生什么"。
- 可以和原时钟项目的"查看某地此刻几点"自然搭配（但仍建议作为独立项目，视情况再决定是否整合）。

## 建议的第一步
不要直接开始写代码。先花时间调研：
1. GDELT 或同类数据源是否能满足"地理位置+事件分级+新闻链接"三个需求。
2. 如果选择方向A，先手动整理10-20条事件数据，做一个静态原型验证地图交互是否有价值。
3. 如果选择方向B，先调研可用的新闻API（按地区/语言过滤的能力、免费额度、调用限制）。

## 一句话目标
做一个"地图上看世界发生了什么"的可视化工具，核心是地理定位准确、分级有意义、交互自然——技术选型应该围绕这三点倒推，而不是先定架构再找数据。
